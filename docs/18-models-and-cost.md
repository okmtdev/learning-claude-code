# 18. モデル選択とコスト戦略

Claude Code は用途に応じてモデルを選べます。**速さ・賢さ・コスト**のバランスを理解し、
「重い思考は賢いモデル、単純作業は安いモデル」と使い分けるのが上達の近道です。

## 1. モデルの段階（ティア）

Claude には用途別のティアがあります（新しい順・このリポジトリ執筆時点）:

| ティア | モデル例 | 向き |
|---|---|---|
| 最上位（賢い・重い） | **Opus 4.8** | 難しい設計・複雑なデバッグ・長い推論 |
| バランス | **Sonnet 4.6** | 日常のコーディング全般（既定にしやすい） |
| 高速・軽量（安い） | **Haiku 4.5** | 定型処理・要約・大量の単純作業 |
| 特殊 | **Fable 5** | 用途特化 |

> 正確なモデル ID や料金は変わります。`/model` の一覧と公式 pricing で最新を確認してください。

## 2. モデルを切り替える

`claude` 起動中:

```
/model            # 一覧から選ぶ
```

headless（[12](12-headless-automation.md)）では:

```bash
claude -p "..." --model sonnet
```

スキル（[15](15-skills.md)）やサブエージェント（[05](05-subagents.md)）では、
フロントマターで**その実行中だけ**モデルを上書きできます:

```yaml
# サブエージェント / スキルの frontmatter
model: haiku     # この役だけ安い高速モデルに
```

`inherit`（既定）は親セッションのモデルを引き継ぎます。

## 3. 使い分けの指針

| 作業 | おすすめ |
|---|---|
| アーキテクチャ設計・難しいバグ調査 | Opus（賢さ優先） |
| 通常の実装・リファクタ・レビュー | Sonnet（バランス） |
| 大量ファイルの単純抽出・整形・要約 | Haiku（速さ・コスト） |
| 調査系サブエージェント（code-explorer 等） | Haiku/Sonnet を割当て |
| 外部下請け（要約・別視点） | Gemini Flash / ローカル Ollama（[10](10-gemini-as-tool.md)） |

**マルチエージェント×モデル割当て**が効きます: メインは Sonnet/Opus、
並列の調査役（[14](14-multi-agent.md)）は Haiku、という構成でコストと速度を最適化できます。

## 4. コストを“見える化”する

- 対話中: `/cost`（あれば）でセッションのおおよその使用量を確認。
- headless: `--output-format json` の `total_cost_usd` で1回ごとの費用を取得（[12](12-headless-automation.md)）。

```bash
claude -p "hello" --output-format json | jq '.total_cost_usd'
```

スクリプトで合算すれば、自動化処理のコストを定量管理できます。

## 5. コストを抑える実践テク

1. **コンテキストを絞る**（[16](16-context-management.md)）: 不要な履歴・巨大ファイルを載せない。
   トークン削減が一番効く。
2. **`/clear` をこまめに**: 話題が変わったらリセット。
3. **安いモデルに下請け**: 単純作業はサブエージェント/スキルで Haiku に。
4. **外部モデルにオフロード**: 巨大入力の要約は Gemini/Ollama へ（[10](10-gemini-as-tool.md)）。
5. **計画モードで手戻り防止**（[03](03-core-features.md)）: 大きな変更は方針確認してから着手。
6. **思考量(effort)の調整**: スキルの `effort: low` などで軽い作業は浅く。

## 6. まとめ

- 「全部 Opus」ではなく、**作業の難度にモデルを合わせる**。
- メイン=賢い/バランス、下請け=安い、の二層構成が基本。
- コンテキスト管理（16章）と組み合わせると、品質を保ったままコストが下がる。
