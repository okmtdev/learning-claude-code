# 09. ループエンジニアリング（自己修正ループ）

「ループエンジニアリング」は公式に1つに定義された用語ではありませんが、実務では主に
**エージェントが自分で軌道修正しながら進む“フィードバックループ”をうまく設計すること** を指します。
あわせて、Claude Code の定期実行機能 `/loop` も紹介します（別物なので区別します）。

## 1. なぜ「ループ」が大事か

LLM エージェントは本質的に次を繰り返して動きます:

```
行動する(ツール実行) → 結果を観察する → 直す → また行動する …
```

このループが**勝手に良い方向へ回る**かどうかは、
**Claude が自分で「正解/不正解」を判定できる手段（＝検証手段）を持っているか**で決まります。

- 検証手段がある（テスト・型チェック・lint・実行結果）
  → Claude は失敗を自分で観測して直せる。**「通るまで直して」の一言で自走**する。
- 検証手段がない
  → Claude は合っているか分からず、人間が毎回フィードバックする必要がある。

つまりループエンジニアリングの核心は **「良いゴール判定をエージェントに渡すこと」** です。
このリポジトリの `sandbox` にはテストがあるので、これをゴール判定に使えます。

## 2. 体験: テストが緑になるまで自走させる 🟢

`sandbox/exercises/` に、**わざとバグを入れた** 練習問題を用意しました。
整数をローマ数字に変換する `toRoman` ですが、減算表記（4=IV, 9=IX, ...）が抜けていて
テストが一部失敗します。

### まず失敗を確認

```bash
cd sandbox && npm run test:exercise
```

`# pass 2 / # fail 2` のように、一部だけ失敗するはずです。

### ループを回す（ターミナル2枚）

**ターミナル A** でモニター:

```bash
./scripts/watch-agents.sh
```

**ターミナル B** の `claude` で、専用コマンドを実行:

```
/fix-until-green
```

または自然言語で同じことを頼めます:

```
sandbox/exercises のテストが全部通るまで、実装を修正→npm run test:exercise→失敗を読む、を繰り返して。
テストファイルは変更しないで、実装側だけ直して。
```

すると Claude が:
1. `npm run test:exercise` を実行（🔧 TOOL がモニターに流れる）
2. 失敗メッセージを読む
3. `toRoman.js` を修正（🔧 TOOL）
4. 再実行 → まだ失敗なら 2 へ…
を **自分で繰り返し**、最終的に全テストを green にします。

モニターに 🔧 が連続して流れるのが、まさに「ループが回っている」様子です。

> 終わったら元に戻すには: `git checkout sandbox/exercises/toRoman.js`

### ここでの学び

- **テストが“ゴール判定”** になっているので、Claude は人間の指示なしで失敗を観測し直せた。
- 「テストを書き換えて通す」のような近道をされないよう、コマンド側で
  「実装だけ直す」と制約を明示している（`.claude/commands/fix-until-green.md`）。
  → **ループのゴールと禁止事項を明確にする**のも設計の一部。

## 3. ループを“良く”する設計のコツ

| やること | 効果 |
|---|---|
| テスト/型/lint を用意する | エージェントが自己採点できる＝自走できる |
| 「通るまで繰り返して」と明示する | 1回で諦めず反復してくれる |
| ゴールと禁止事項を書く | テスト改変などの近道を防ぐ |
| 計画モード（[03](03-core-features.md)）で着手前に方針確認 | 大きな手戻りを防ぐ |
| 1ループを小さく保つ | 観察→修正のサイクルが速くなる |

> CLAUDE.md に「テストは `cd sandbox && npm test`」と書いてあるのも、
> エージェントがループの検証コマンドに迷わないための“足場”です。

## 4. もう一つの「ループ」: `/loop` 機能（定期実行）

`/loop` は、プロンプトやスラッシュコマンドを**一定間隔で繰り返し実行**する機能です。
2 章の「自己修正ループ」とは別物で、こちらは **“繰り返し起動”** の話です。

使い方の例（`claude` 起動中）:

```
/loop 5m sandbox のテストを実行して、失敗があれば直して
```

- `5m` = 5分間隔（省略時は既定値）。
- CI 的な監視や、定期的なチェックを回したいときに使います。
- 止めたいときは指示すれば停止します。

> 用途の違い:
> - **自己修正ループ（2章）**: 1つのタスクを“完了するまで”内部で反復。
> - **`/loop`（本節）**: 同じ指示を“時間間隔で”何度も起動。

## 5. 応用アイデア

- `sandbox` に新しい関数の**テストだけ先に書かせて**、その後「テストが通るまで実装して」と頼む（テスト駆動 × ループ）。
- [`/review-diff`](04-slash-commands.md) でレビュー → 指摘を直す → 再レビュー、をループにする。
- [Postgres MCP（08）](08-mcp-postgres.md) で「クエリを実行 → 結果を見て条件を直す → 再実行」も同じループ構造。
