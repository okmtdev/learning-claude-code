# 16. コンテキスト管理（コンテキストエンジニアリング）

Claude Code の出力品質は、**コンテキスト（文脈）に何が入っているか**で大きく変わります。
長い作業で迷子にさせない・トークンを浪費しない・必要な情報を確実に渡す——その設計を学びます。

## 1. コンテキストは有限の「作業机」

モデルが一度に見られる情報量（コンテキストウィンドウ）には上限があります。
机に資料を広げすぎると、重要なものが埋もれて精度が落ちます。原則:

- **必要なものを、必要なときだけ載せる**（不要な履歴は減らす）。
- **恒久的な前提は CLAUDE.md に固定**（毎回説明しない）。
- **一時的な参照は @ で都度添付**（履歴に残し続けない）。

## 2. CLAUDE.md = 常駐の前提

`CLAUDE.md` はセッション開始時に自動で読まれる「プロジェクト取扱説明書」です。
このリポジトリの `CLAUDE.md` を見てください — 目的・ディレクトリ・テスト方法・禁止事項が書いてあります。

良い CLAUDE.md:
- **短く・要点だけ**（スキルや reference に詳細を逃がす）。
- コマンド（テストの回し方）、規約、触ってはいけない場所を明記。
- 階層化できる: ルートの `CLAUDE.md` ＋ サブディレクトリ用の `CLAUDE.md`。

更新の小技:

```
#このリポジトリのテストは sandbox ディレクトリで `npm test`
```

`#` で始めると、その内容を CLAUDE.md などのメモに追記できます。
自動生成は `/init`。

## 3. 入力を絞る道具

| 操作 | 効果 |
|---|---|
| `@path/to/file` | そのファイルだけ文脈に添付（補完が効く） |
| `#メモ` | 恒久メモとして記憶（CLAUDE.md 等へ） |
| 計画モード(Plan) | 読み取りのみで方針を固める（[03](03-core-features.md)） |
| サブエージェント | 調査を別文脈に隔離して結論だけ受け取る（[14](14-multi-agent.md)） |
| スキルの段階的開示 | 詳細を別ファイルにし、必要時のみロード（[15](15-skills.md)） |

### 試す: 必要なファイルだけ渡す

```
@sandbox/src/mathUtils.js この average の空配列時の挙動だけ説明して
```

リポジトリ全体を読ませるより速く・正確で、文脈も汚しません。

## 4. 不要になった文脈を片付ける

長く会話すると、過去のやり取りが机を圧迫します。

| コマンド | 効果 | 使いどころ |
|---|---|---|
| `/clear` | 会話履歴をリセット | **話題が変わったら**まず実行。混線を防ぐ最強の一手 |
| `/compact` | 履歴を要約して圧縮 | 文脈は保ちたいが長くなりすぎた時 |
| `/context`（あれば） | 現在のトークン使用状況を確認 | 重くなってきたら点検 |

> 目安: 「別のタスクを始める」なら `/clear`。
> 「同じタスクだが会話が長い」なら `/compact`。

## 5. サブエージェント / スキルでの“文脈の隔離”

- **サブエージェント**は独立した文脈を持ちます。巨大なログ調査や広い探索を任せると、
  メインの机は結論だけで済みます（[14](14-multi-agent.md)）。
- **`context: fork` のスキル**も同様に独立実行できます（[15](15-skills.md) の `explain-arch`）。
- **巨大入力は外部モデルへ**: 長大ファイルの要約を Gemini に逃がす手も（[10](10-gemini-as-tool.md)）。

## 6. 実践のコツ（まとめ）

1. タスクが変わったら `/clear`。
2. 恒久前提は CLAUDE.md、一時参照は `@`。
3. 広い調査はサブエージェント/forkスキルに隔離。
4. 長くなったら `/compact`。
5. 1メッセージ1論点。情報を盛りすぎない。

> コンテキスト管理は「魔法のプロンプト」より効く、最も費用対効果の高い改善です。
