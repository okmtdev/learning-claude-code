# 14. マルチエージェント・オーケストレーション

[05](05-subagents.md) で1体のサブエージェントを動かしました。本章では **複数のサブエージェントを
分業・並列** で動かし、メインが**統合役**になる構成を学びます。Monitor が最も映える章です。

## なぜ複数に分けるのか

- **並列で速い**: 独立した調査を同時に走らせれば待ち時間が短い。
- **文脈が汚れない**: 各エージェントが自分の担当だけに集中。メインの会話を圧迫しない。
- **役割で品質UP**: 「実装役」と「レビュー役」を分けると、批判的な視点が入る。

## 代表的なパターン

| パターン | 形 | 例 |
|---|---|---|
| **fan-out → 統合** | 複数が並列調査 → 1人がまとめる | リポジトリ多角監査 |
| **パイプライン** | 調査 → 実装 → レビュー と順に渡す | 機能追加の一連 |
| **分割統治** | 大きな対象を範囲で分担 | 大量ファイルの解析 |

## このリポジトリのエージェント

- `code-explorer` … 読み取り専用の調査役（[05](05-subagents.md)）
- `test-runner` … テスト実行役
- `reviewer` … 批判的レビュー役（本章で追加）

`claude` 起動中に `/agents` で一覧を確認できます。

## 🎬 デモ: 並列監査 `/parallel-audit`

**ターミナル A** でモニター:

```bash
./scripts/watch-agents.sh
```

**ターミナル B** の `claude` で:

```
/parallel-audit
```

これは **fan-out → 統合** パターンです。4つの調査を並列起動し、最後にメインが統合します:

1. `code-explorer`: sandbox/src の関数一覧
2. `code-explorer`: コマンド/エージェントの棚卸し
3. `reviewer`: textUtils / mathUtils のレビュー
4. `test-runner`: テスト実行

モニターには 🚀 SPAWN が立て続けに4つ → それぞれ ✅ DONE が流れ、
**複数エージェントが本当に並列で動いている**様子が見えます。
最後にメインが統合レポート（要点・矛盾・次の一手）を返します。

## パイプライン（調査 → レビュー）を手で組む

自然言語でも分業を指示できます:

```
まず code-explorer で sandbox/src/mathUtils.js の各関数の責務を洗い出して。
その結果を踏まえて reviewer に同ファイルをレビューさせて。
最後にあなた(メイン)が、両者を突き合わせて改善タスクを3つに整理して。
```

「調査の出力」を「レビューの入力」に渡す＝パイプラインです。

## 知っておくべき制約

- **並列実行**: メインは複数サブエージェントを同時に起動できます（1メッセージ内で複数 Task）。
- **ネストの深さ上限は 5**: サブエージェントはさらに子を産めますが、メインから数えて最大5階層。
  最下層は子を産めません。無限に増殖はしません。
- **読み取り専用エージェントを活用**: 調査・レビューは `tools: Read, Grep, Glob` に絞ると安全
  （誤った変更を防げる）。実装は別の権限を持つエージェント/メインで行う。

## コストと使いどころ

- エージェントを増やすほどトークン消費は増えます。**「並列で速くしたい」「別視点が欲しい」時に使う**。
- 単純な1ファイル修正など、分ける意味が薄い作業は1体（やメイン）で十分。
- 安価なモデルを下請けに割り当てる発想は [10](10-gemini-as-tool.md) や `/model` と組み合わせられます。

## 自分のチームを足してみる

例えば「ドキュメント整形役」を `.claude/agents/doc-writer.md` として追加し、
`/parallel-audit` に1工程足す、といった拡張ができます。役割・`tools`・`description` を
明確に書くのがコツです（[05](05-subagents.md) のフロントマター参照）。
